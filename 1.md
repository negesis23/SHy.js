### 1. Bisakah Proxy disatukan ke dalam `s` (Satu API)?
**Bisa.** Ini adalah rute yang diambil oleh Vue dengan `ref()`. 
Di dalam fungsi `s(initialValue)`, kita tinggal melakukan pengecekan: jika `typeof initialValue === 'object'`, kita kembalikan versi yang dibungkus Proxy. Jika primitif (string, number), kita kembalikan signal biasa.

**Tantangan Paradigma (DX):**
Signal Anda saat ini mengembalikan *tuple* `[getter, setter]`.
*   Jika primitif: `const [count, setCount] = s(0); setCount(1);`
*   Jika objek dengan Proxy di dalamnya: `const [user, setUser] = s({ name: 'A' });`

Bagaimana pengguna mengubah namanya?
1.  **Gaya Mutasi (Vue-style):** `user().name = 'B'`. (Memanfaatkan Proxy *intercept* pada *getter*). Ini mengorbankan kegunaan `setUser`, karena state langsung dimutasi.
2.  **Gaya Path (Solid-style):** `setUser('name', 'B')`. (Setter dimodifikasi untuk menerima *path* objek).

Jika Anda ingin satu API saja, kita harus memilih salah satu DX di atas agar pengguna tidak bingung kapan harus pakai `setUser` dan kapan harus langsung merubah properti objek.

### 2. Proxy itu lambat, haruskah dipakai?
Anda 100% benar. Jika kita menggunakan Proxy untuk **menyimpan status** atau melacak ketergantungan di dalamnya, performanya akan hancur. 

**Solusi Arsitektur yang Tepat:**
Seperti kata Anda, Proxy **HANYA** boleh digunakan sebagai "membran" atau "pembungkus transparan" (*transparent interceptor*). 
*   Data aslinya tetaplah *plain object* biasa.
*   Logika *dependency tracking* (grafik reaktivitas) hidup sepenuhnya **di luar** objek tersebut (di mesin *core* kita).
*   Saat `proxy.nama` dibaca, Proxy hanya meneriakkan *"Hei Core, fungsi yang sedang berjalan butuh properti 'nama'!"*, lalu langsung me-return nilai aslinya. Ia tidak menyimpan *subscriber* di dalam dirinya.

### 3. Mengapa tidak `defineProperty` saja?
Vue 2 menggunakan `Object.defineProperty` (getter/setter murni) dan memang lebih cepat untuk membaca properti yang sudah ada. 

**Kelemahannya:** `defineProperty` tidak bisa mendeteksi jika pengguna menambahkan properti *baru* yang belum ada saat inisialisasi (misal: `state().hobiBaru = 'Mancing'`) atau ketika pengguna memanipulasi *index array* (`state().list[10] = 'X'`). Untuk *fine-grained reactivity* modern yang mendukung penambahan properti dinamis, Proxy (sebagai pembungkus sangat tipis) adalah kejahatan yang terpaksa (*necessary evil*), meskipun sedikit lebih lambat dari *getter/setter* murni.

### 4. `new Set` kalah cepat dari Linked List
Lagi-langkah Anda sangat *up-to-date*. SolidJS 2.0 baru-baru ini menulis ulang seluruh mesin reaktivitas mereka: membuang struktur array/`Set` untuk melacak *subscriber*, dan beralih sepenuhnya ke **Doubly Linked List**. 

Alasannya persis seperti yang Anda bilang: menambahkan dan menghapus node (efek) dari *linked list* adalah operasi **O(1)** yang tidak memicu alokasi memori tambahan atau membebani *Garbage Collector*, berbeda dengan `Set` yang lambat saat membesar dan menyusut ribuan kali per detik.

*   **Dengan Proxy Membran:** Saat komponen memanggil `user().name`, Proxy akan mencegat (*intercept*) panggilan itu.

    *   Proxy mendeteksi `get` pada properti `age`.

### Bagaimana Auto-Batching Bekerja Secara Otomatis?

Jika Anda memanggil beberapa setter berturut-turut:
```javascript
setCount(1);
setUser({ name: 'Budi' });
setTheme('dark');
```

Alih-alih langsung menjalankan ulang (re-render) komponen setiap kali `set` dipanggil (yang akan menyebabkan 3 kali re-render yang sia-sia), mesin reaktivitas kita melakukan taktik **"Tunda dan Kumpulkan"**:

1. **Pengumpulan (Queueing):**
   Saat `setCount(1)` dipanggil, mesin tidak langsung mengeksekusi efek/komponen. Mesin hanya memasukkan efek tersebut ke dalam sebuah daftar tunggu (misalnya `pendingEffects.add(effect)`).
2. **Penjadwalan (Scheduling):**
   Mesin menyalakan sebuah "alarm" untuk berjalan di akhir putaran eksekusi saat ini menggunakan `Promise.resolve().then(...)` (ini disebut *Microtask*).
3. **Pengumpulan Lanjutan:**
   Saat `setUser` dan `setTheme` dipanggil di baris berikutnya, alarm sudah menyala. Mesin hanya terus menambahkan efek-efek yang terpengaruh ke dalam daftar tunggu yang sama. Karena menggunakan `Set` (atau penanda unik di Linked List nantinya), efek yang sama tidak akan dimasukkan dua kali meskipun dipicu berkali-kali.
4. **Eksekusi Sekaligus (Flushing):**
   Setelah semua kode sinkron Anda selesai dieksekusi, JavaScript Engine (V8) akan melihat antrean Microtask. Alarm kita berbunyi! Mesin reaktivitas mengambil seluruh daftar `pendingEffects`, menjalankannya **hanya satu kali**, dan kemudian mengosongkan daftarnya.

### Bukti di dalam Kode SHy.js Kita Saat Ini
Jika Anda melihat file `core.ts` yang sudah ada, sistem auto-batching pintar ini sebenarnya sudah berjalan:

```typescript
// Di dalam fungsi notifyEffect (saat setter dipanggil)
pendingEffects.add(subscriber); // 1. Kumpulkan efek

if (!microtaskQueued) {
  setMicrotaskQueued(true);
  
  // 2. Pasang alarm microtask
  Promise.resolve().then(() => {
    setMicrotaskQueued(false);
    
    // 4. Eksekusi semua sekaligus (Batch otomatis!)
    for (const effToRun of pendingEffects) {
      runInErrorHandler(() => effToRun());
    }
    pendingEffects.clear();
  });
}
```

### Jika Sudah Otomatis, Kenapa Framework Lain Punya `batch()`?

Framework seperti SolidJS atau React terkadang masih mengekspos fungsi `batch()` untuk dua alasan spesifik, bukan untuk batching dasar:

1. **Opt-out dari Microtask:** Beberapa framework memilih mengeksekusi efek secara sinkron murni (tanpa `Promise.resolve()`) untuk alasan performa atau agar tumpukan error (*stack trace*) lebih mudah dibaca. Jika frameworknya sinkron murni, maka mereka *wajib* punya `batch()` agar pengembang bisa menggabungkan re-render secara manual.
2. **Asynchronous Batching:** Jika Anda melakukan `await fetch(...)`, lalu ada banyak `set` setelahnya, terkadang kontrol eksekusi menjadi terpecah. Namun di JavaScript modern, auto-batching microtask umumnya sudah cukup pintar untuk menangani hampir semua kasus harian.

**Kesimpulan:**
Dengan arsitektur berbasis Microtask yang kita miliki saat ini, Anda **tidak perlu** membuat API `batch(() => {})`. Pengguna bisa memanggil ratusan `setter` berurutan dalam satu fungsi, dan SHy.js dengan pintar hanya akan melakukan re-render **satu kali** di akhir proses. DX menjadi jauh lebih bersih!

Berikut adalah alasan teknis mengapa kita harus langsung menggantinya sekarang:

### 1. Masalah Utama pada `Set` (Garbage Collection Thrashing)
Dalam sistem *fine-grained reactivity* (seperti yang baru kita buat dengan Proxy), komponen/efek sering sekali **berhenti berlangganan (unsubscribe)** dan **berlangganan ulang (resubscribe)** setiap kali mereka dijalankan ulang (re-run). 
*   Saat menggunakan `Set`, setiap kali elemen ditambahkan atau dihapus, *engine* V8 JavaScript harus mengatur ulang ukuran (resize) struktur memori di bawah kap mesin.
*   Jika ada ribuan perubahan per detik (misalnya animasi atau iterasi list panjang), alokasi memori ini akan memicu **Garbage Collector (GC)** bekerja keras, yang menyebabkan *lag* atau *stuttering* pada UI (FPS drop).

### 2. Keunggulan Mutlak Doubly Linked List (O(1) Murni)
Dengan Doubly Linked List, kita tidak menggunakan struktur data bawaan yang berat. Kita hanya menambahkan properti `prev` (sebelumnya) dan `next` (selanjutnya) langsung ke dalam objek/fungsi efek itu sendiri.

*   **Menambah subscriber:** Tinggal ubah *pointer* `next` dari *node* terakhir ke *node* baru. **(Kecepatan kilat, nol alokasi memori baru).**
*   **Menghapus subscriber:** Tinggal sambungkan *node* `prev` dan `next` dari *node* yang dihapus, melewatinya. **(Sangat instan, tidak ada array/set yang perlu digeser atau di-resize).**

### 3. Mengapa SolidJS 2.0 Rela Menulis Ulang Core Mereka?
Ryan Carniato (Pencipta SolidJS) menyadari bahwa pada skala ekstrem, operasi iterasi dan pembersihan `Set` menjadi *bottleneck* (leher botol) performa. Dengan pindah ke eksekusi berbasis grafik (*Graph-based execution*) menggunakan pointer Linked List, penggunaan memori turun drastis dan kecepatan eksekusi meningkat signifikan.

1.  **Struktur Node:** `EffectFn` (yang saat ini hanya fungsi) akan diperluas menjadi objek atau memiliki properti tersembunyi seperti `effect._next`, `effect._prev`.
2.  **Pembuangan `Set`:** Kita akan membuang `pendingEffects = new Set()`, `effectDependencies = new WeakMap<... Set...>`, dll.
3.  **Pengganti:** Kita akan menggantinya dengan variabel penunjuk (*head* dan *tail* pointer).
